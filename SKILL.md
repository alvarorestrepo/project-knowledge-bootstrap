## Purpose

You are a sub-agent responsible for KNOWLEDGE GENERATION. You analyze a project's
codebase and produce domain knowledge artifacts that eliminate the AI's cold-start
problem. You are NOT an orchestrator — you generate knowledge, SDD manages process.

**What you produce:**

1. **AGENTS.md** — A lightweight map (~100-150 lines) at project root
2. **Domain skills** — Deep knowledge per module in `skills/` directory
3. **Infrastructure** — `setup.sh` (multi-env installer) + `skill-sync/` (auto-invoke sync)

**What you do NOT do:**

- Manage workflow (that's SDD's job)
- Create openspec directories or SDD artifacts
- Make architectural decisions about the project
- Modify existing source code

## Phase 0: Pre-Flight — Detect Existing Artifacts

Before doing ANYTHING else, check what already exists in the project. This
determines whether you operate in **CREATE MODE** (fresh generation) or
**UPDATE MODE** (smart merge with existing artifacts).

### Step 0.1: Validate Project

Confirm this is a real project (has source code, not an empty directory):
- Check for at least one of: package.json, go.mod, pyproject.toml, Cargo.toml,
  pom.xml, build.gradle, Makefile, CMakeLists.txt, *.sln, mix.exs, Gemfile
- If NONE found, warn the user that this may not be a project root

### Step 0.2: Check for Existing AGENTS.md

Search project root for:
```
├── AGENTS.md
├── CLAUDE.md
├── GEMINI.md
├── .github/copilot-instructions.md
└── skills/ directory
```

**Detection logic:**
1. If `AGENTS.md` exists, read it and look for the signature comment:
   ```
   <!-- Generated by project-knowledge-bootstrap v{X.Y} -->
   ```
2. **If signature IS found** → this file was generated by this skill.
   Enter **UPDATE MODE** automatically.
3. **If signature is NOT found** (user-created AGENTS.md) → **WARN** the user:
   > "⚠ This project has an AGENTS.md that was NOT generated by this skill
   > ({line_count} lines). Options:
   > (a) Back up existing and regenerate from scratch (CREATE MODE)
   > (b) Abort — do not modify anything"
   Wait for user response. Do NOT proceed without confirmation.
4. **If AGENTS.md does NOT exist** → Enter **CREATE MODE**.

### Step 0.3: Check for Existing skills/ Directory

If `skills/` directory exists:

1. **Inventory all existing skills** — list each `skills/*/SKILL.md`
2. **Classify each skill** by reading its frontmatter:
   - If it contains `generated_by: project-knowledge-bootstrap` in the
     `metadata` section → **GENERATED skill** (safe to update)
   - If it does NOT contain this field → **HAND-WRITTEN skill** (NEVER touch)
3. Record which modules already have generated skills

### Step 0.4: Check for Existing setup.sh and sync.sh

If `skills/setup.sh` or `skills/skill-sync/assets/sync.sh` exist:

1. Read the version comment at the top of each file:
   ```bash
   # Generated by project-knowledge-bootstrap v1.0
   # Version: 1.0
   ```
2. Compare the `Version:` line with the current template version
3. **Only update if the template version is newer** — otherwise leave as-is
4. If no version comment is found, treat as user-modified and WARN before replacing

### Step 0.5: Determine Mode and Report

Before proceeding, report the detection results:

```
## Pre-Flight Results

**Mode**: CREATE | UPDATE
**Existing artifacts found**:
- AGENTS.md: {exists/missing} {generated/user-created}
- skills/ directory: {exists/missing} ({N} generated, {M} hand-written)
- setup.sh: {exists/missing} (version: {X.Y} | no version | missing)
- sync.sh: {exists/missing} (version: {X.Y} | no version | missing)

Proceeding in {MODE} mode...
```

### CREATE MODE vs UPDATE MODE Reference

| Aspect | CREATE MODE | UPDATE MODE |
|--------|-------------|-------------|
| AGENTS.md | Generate fresh from template | Smart merge (see below) |
| Generated skills | Generate fresh | Re-analyze module, update content |
| Hand-written skills | N/A | NEVER touch |
| New modules | Generate skills | Generate skills |
| Removed modules | N/A | WARN user, do NOT delete skill |
| setup.sh | Generate from template | Update only if template version newer |
| sync.sh | Generate from template | Update only if template version newer |

### UPDATE MODE: Smart Merge Rules for AGENTS.md

When updating an existing generated AGENTS.md:

1. **`## Stack` section**: REPLACE entirely — re-detected from current project state
2. **`## Architecture` section**: REPLACE entirely — re-detected
3. **`## Modules` table**:
   - **Existing modules still present** → KEEP the row, update description if changed
   - **New modules detected** → ADD new rows
   - **Modules no longer found** → Do NOT remove. Add a comment: `<!-- MODULE REMOVED? Verify and delete manually -->`
4. **`## Available Skills` table**: REGENERATE based on current skills inventory
5. **`## Auto-invoke Skills` table**: Always regenerated by sync.sh (expected behavior)
6. **`## Critical Rules` section**:
   - **Rules that match existing rules** → KEEP as-is (user may have refined wording)
   - **New rules detected from analysis** → APPEND at end with marker: `<!-- Added by bootstrap update {date} -->`
   - **NEVER remove existing rules** — user may have added custom ones
7. **`## Development` section**: REPLACE with re-detected commands
8. **Signature comments**: Update the `<!-- Last updated: ... -->` timestamp

### UPDATE MODE: Skill Update Rules

For each generated skill:

1. Re-analyze the module (directory structure, patterns, tests may have changed)
2. REPLACE the skill content entirely — generated skills are fully managed
3. Update the `generated_at` timestamp in frontmatter
4. Preserve the skill's file path (don't rename)

For hand-written skills:
- **NEVER modify, rename, or delete** — these are user-owned
- Include them in the Available Skills table and Auto-invoke sync

For new modules without skills:
- Generate new skills following Phase 3

For modules that no longer exist:
- Do NOT delete the skill directory
- WARN in the Bootstrap Report: "Module {name} no longer found at {path}. Skill preserved. Delete manually if no longer needed."

## Phase 1: Deep Analysis

Analyze the project systematically. Do NOT skip any step.

### Step 1.1: Detect Tech Stack

```
Scan for (in order):
├── package.json       → Node.js ecosystem (check for framework in dependencies)
│   ├── next           → Next.js (check version)
│   ├── react          → React (check version)
│   ├── @angular/core  → Angular (check version)
│   ├── vue            → Vue.js (check version)
│   ├── svelte         → Svelte/SvelteKit
│   └── express/fastify/hono → Backend framework
├── go.mod             → Go (check module path, Go version)
├── pyproject.toml     → Python (check build system, dependencies)
│   ├── django         → Django (check version)
│   ├── fastapi        → FastAPI
│   ├── flask          → Flask
│   └── poetry/hatch   → Package manager
├── Cargo.toml         → Rust (check edition, dependencies)
├── pom.xml            → Java/Maven
├── build.gradle       → Java/Kotlin/Gradle
├── mix.exs            → Elixir
├── Gemfile            → Ruby
├── *.sln / *.csproj   → .NET/C#
└── CMakeLists.txt     → C/C++

Also check:
├── .nvmrc / .node-version    → Node version constraint
├── .python-version           → Python version constraint
├── .tool-versions            → asdf version manager
├── Dockerfile / compose.yaml → Containerization
└── Makefile                  → Build automation
```

Record: `{language}`, `{framework}`, `{version}`, `{package_manager}`.

### Step 1.2: Detect Architecture Pattern

```
Analyze directory structure:
├── Multiple package.json / go.mod / pyproject.toml at different levels?
│   └── MONOREPO (record workspace config)
├── Separate directories with their own servers (api/, ui/, worker/)?
│   └── FULLSTACK MONOLITH or POLYREPO-IN-MONO
├── Single src/ with clear layer separation (controllers/, services/, repos/)?
│   └── LAYERED ARCHITECTURE
├── cmd/ + internal/ + pkg/ (Go convention)?
│   └── GO STANDARD LAYOUT
├── src/ + lib/ + tests/ with single entry point?
│   └── LIBRARY
├── apps/ + packages/ with workspace config?
│   └── MONOREPO (Turborepo/Nx/Lerna)
└── Flat structure with index.* entry?
    └── SIMPLE APP or SCRIPT
```

For monorepos, identify EACH component and its role.

### Step 1.3: Detect Module Structure

For each top-level directory that represents a module/component:

```
Record:
├── Path (relative to root)
├── Purpose (inferred from name + contents)
├── Tech stack (may differ per module in monorepo)
├── Entry point(s)
├── Has own tests? (check for test/, __tests__/, *_test.*, *.spec.*)
├── Has own config? (tsconfig, pyproject, etc.)
└── Key patterns visible in file structure
```

**Module detection heuristics:**
- Has its own package.json/go.mod/pyproject.toml → independent module
- Has its own README → documented module
- Contains src/ or lib/ → code module (not config/docs)
- Named services/, providers/, checks/ → domain module
- Named .github/, .husky/, scripts/ → infrastructure (not a domain module)

**Grouping rule**: If there are >8 modules, GROUP related ones. The AGENTS.md
must stay under ~150 lines. Group by:
- Domain similarity (all `providers/aws*` → "AWS Provider Checks")
- Functional role (all test directories → "Testing Infrastructure")
- Tech similarity (all Python services → "Python Services")

Check for each:

│   ├── *_test.go                       → Go tests
│   └── playwright.config*              → E2E tests
├── CI/CD
│   ├── .github/workflows/             → GitHub Actions

### Step 1.5: Detect Git Conventions

```bash
# Check recent commit messages (10-20) for patterns
git log --oneline -20

# Detect format:
# - Conventional Commits: "feat(scope): description"
# - Gitmoji: ":sparkles: description"
# - Free-form: no pattern
# - Issue-linked: "#123 description" or "JIRA-123 description"

# Check branch naming
git branch -a | head -20

# Detect branching strategy:
# - GitFlow: develop, release/*, feature/*, hotfix/*
# - GitHub Flow: main + feature branches
# - Trunk-based: main only
```

Look for patterns in the codebase that indicate MUST/NEVER rules:

├── CI workflow files → Required checks
├── Existing AGENTS.md / CLAUDE.md → Existing AI rules
├── README.md → Project-level constraints
├── Architecture Decision Records (ADR) → Design decisions

## Phase 2: Generate AGENTS.md

Use the template from `assets/AGENTS-TEMPLATE.md`. The generated file MUST:

- Be under 150 lines (aim for 100-120)
- Use REAL data from Phase 1 analysis (never placeholder text)
- Have a complete Auto-invoke table (generated by sync.sh later, but include manual entries found)
- Include **signature comments** (see below)

**If in UPDATE MODE**: Follow the Smart Merge Rules from Phase 0 instead of
generating from scratch. Read the existing AGENTS.md, apply merge logic per
section, and update the timestamp.

### Signature Comments (REQUIRED)

Every generated AGENTS.md MUST include these comments at the very top of the
file, BEFORE the `# {Project Name}` heading:

```markdown
<!-- Generated by project-knowledge-bootstrap v1.0 -->
<!-- Last updated: {ISO 8601 date, e.g. 2026-03-17T12:00:00Z} -->
```

These markers enable idempotent re-runs by allowing Phase 0 to detect
whether an AGENTS.md was generated by this skill.

### AGENTS.md Structure

```markdown
<!-- Generated by project-knowledge-bootstrap v1.0 -->
<!-- Last updated: {ISO date} -->

# {Project Name}

## Stack
(3-5 lines max — only the KEY technologies)

## Architecture
(2-3 lines — pattern name + key directory layout)

## Modules
(Table format — module | path | description | skill reference)

## Available Skills
(Table: skill name | description | SKILL.md link)

## Auto-invoke Skills
(Table: action | skill — ALWAYS load skill FIRST for these actions)
<!-- Generated by skills/skill-sync/assets/sync.sh — DO NOT edit manually -->

## Critical Rules
(5-10 bullet points — the MUST/NEVER rules of this project)
```

### Writing Rules for AGENTS.md

1. **Stack section**: List ONLY the primary technologies. Not every devDependency.
   - GOOD: "Python 3.11, Django 5.1, DRF, PostgreSQL, Celery, Redis"
   - BAD: Listing every pip package

2. **Architecture section**: Name the pattern and show the TOP-LEVEL layout.
   - GOOD: "Monorepo with 4 components: SDK (Python), API (Django), UI (Next.js), MCP Server"
   - BAD: Describing every subdirectory

3. **Modules table**: One row per module/component. If >8, group them.

4. **Critical Rules**: Extract from analysis. These are the NON-NEGOTIABLE rules.
   - ALWAYS rules: things that must happen every time
   - NEVER rules: things that are forbidden
   - Format rules: commit style, branch naming, etc.
   - Test rules: what must be tested
   - Security rules: auth patterns, data handling

## Phase 3: Generate Domain Skills

For each detected module (or group of modules), generate a skill.

**If in UPDATE MODE**:
- Re-analyze each module and update its generated skill content
- NEVER touch hand-written skills (those without `generated_by` in frontmatter)
- Create new skills for newly detected modules
- Do NOT delete skills for removed modules — warn in the report instead

### Generated-By Frontmatter (REQUIRED)

Every generated SKILL.md MUST include these fields in its YAML frontmatter
`metadata` section:

```yaml
metadata:
  generated_by: project-knowledge-bootstrap
  generated_at: "{ISO 8601 date, e.g. 2026-03-17T12:00:00Z}"
  source_version: "1.0"
```

These markers enable Phase 0 to distinguish generated skills from hand-written
ones during UPDATE MODE. The `generated_at` field is updated on every re-run.

### Step 3.1: Overview Skill

ALWAYS generate `skills/{project}-overview/SKILL.md`:

```yaml
---
name: {project}-overview
description: >
  Global architecture, conventions, and patterns for {Project Name}.
  Trigger: When working on cross-cutting concerns, understanding project
  architecture, or needing general context about {project}.
metadata:
  author: user
  version: "1.0"
  scope: [root]
  generated_by: project-knowledge-bootstrap
  generated_at: "{ISO date}"
  source_version: "1.0"
  auto_invoke:
    - "General {project} development questions"
    - "Understanding {project} architecture"
allowed-tools: Read, Glob, Grep
---
```

Content for overview skill:
- Architecture overview with directory tree
- Component interaction diagram (text-based)
- Global conventions (naming, formatting, testing)
- Common commands (build, test, lint, deploy)
- Environment setup requirements
- Key environment variables
- Database/service dependencies

### Step 3.2: Module Skills

For EACH significant module, generate `skills/{project}-{module}/SKILL.md`:

```yaml
---
name: {project}-{module}
description: >
  Deep knowledge for the {module} module of {Project Name}.
  Trigger: When working on {module}, creating/modifying files in {module_path}/,
  or debugging {module}-related issues.
metadata:
  author: user
  version: "1.0"
  scope: [root]
  generated_by: project-knowledge-bootstrap
  generated_at: "{ISO date}"
  source_version: "1.0"
  auto_invoke:
    - "Working on {module} code"
    - "Creating/modifying files in {module_path}/"
    - "Debugging {module} issues"
allowed-tools: Read, Glob, Grep
---
```

**Each module skill MUST contain** (use content from ACTUAL analysis, not generic text):

1. **Directory Structure** — Real tree output of the module
   ```
   {module_path}/
   ├── src/
   │   ├── controllers/     # Request handlers
   │   ├── services/        # Business logic
   │   └── repositories/    # Data access
   ├── tests/
   └── package.json
   ```

2. **Key Patterns** — Real patterns found in the code
   ```markdown
   ## Key Patterns

   ### Controller Pattern
   All controllers in this module follow:
   - Located in `src/controllers/`
   - Named `{resource}.controller.ts`
   - Use dependency injection via constructor
   - Return DTOs, never raw entities

   ### Example (from actual codebase):
   ```{language}
   // From {actual_file_path}
   {real_code_snippet_showing_pattern}
   ```

3. **Decision Trees** — For common choices in this module
   ```markdown
   ## Decision Trees

   ### Adding a New Endpoint
   1. Create controller in `src/controllers/`
   2. Create service in `src/services/`
   3. Add route in `src/routes/index.ts`
   4. Add validation schema in `src/schemas/`
   5. Add tests in `tests/`
   ```

4. **Critical Rules** — Module-specific ALWAYS/NEVER rules
   ```markdown
   ## Critical Rules

   - ALWAYS use the repository pattern for database access
   - NEVER import from other modules directly (use shared interfaces)
   - ALWAYS add validation schemas for request bodies
   - NEVER expose internal IDs in API responses
   ```

5. **Testing Strategy** — How tests work in this module
   ```markdown
   ## Testing

   - Framework: {detected_test_framework}
   - Location: `{test_directory}/`
   - Naming: `{naming_convention}`
   - Run: `{test_command}`

   ### Test Pattern
   ```{language}
   // From {actual_test_file}
   {real_test_snippet_showing_pattern}
   ```

6. **Common Commands** — Module-specific commands
   ```markdown
   ## Commands

   ```bash
   # Run module tests
   {actual_test_command}

   # Lint module
   {actual_lint_command}

   # Build module
   {actual_build_command}
   ```

### Step 3.3: Skill Content Rules

**CRITICAL — these are NON-NEGOTIABLE:**

1. **Use REAL file paths** from the analyzed project. Not `src/example/`.
2. **Use REAL code snippets** from the codebase. Extract actual patterns, not invented examples.
3. **Use REAL commands** that work in this project. Test them if possible.
4. **Keep each skill under 300 lines**. Focus on what the AI NEEDS to know, not everything.
5. **Include the frontmatter** with proper `scope` and `auto_invoke` fields.
6. **Reference actual files** that implement the patterns you describe.
7. If a module has sub-modules, create ONE skill that covers all of them with sections.

### Step 3.4: When to Skip Module Skills

Do NOT create skills for:
- Infrastructure directories (`.github/`, `.husky/`, `scripts/` unless complex)
- Configuration files (these go in the overview skill)
- Empty directories or placeholder modules
- Modules with <5 files (mention in overview skill instead)
- Third-party/vendored code

## Phase 4: Generate Infrastructure

**If in UPDATE MODE**: Only update setup.sh and sync.sh if the template version
is newer than the installed version. Compare the `# Version: X.Y` comment in
the existing file with the current template version. If versions match or the
existing is newer, skip the update.

### Version Comments (REQUIRED)

Every generated setup.sh and sync.sh MUST include these comments immediately
after the shebang line:

```bash
# Generated by project-knowledge-bootstrap v1.0
# Version: 1.0
```

These markers enable Phase 0 to detect the installed version and decide
whether an update is needed.

### Step 4.1: Create setup.sh

Generate `skills/setup.sh` using the template from `assets/setup.sh`.

Adapt the template for this specific project:
- Replace project name references
- Ensure AGENTS.md paths match the project's module structure
- For monorepos: handle per-component AGENTS.md files
- Verify the script works with Bash 3 (macOS default)

The setup.sh must support these environments:

| Flag | Environment | Skills Directory | Rules File |
|------|-------------|-----------------|------------|
| `--claude` | Claude Code | `.claude/skills/` | `CLAUDE.md` |
| `--cursor` | Cursor | `.cursor/skills/` | `.cursorrules` |
| `--opencode` | OpenCode | `.opencode/skills/` | — (uses AGENTS.md natively) |
| `--gemini` | Gemini CLI | `.gemini/skills/` | `GEMINI.md` |
| `--copilot` | GitHub Copilot | — (no skills dir) | `.github/copilot-instructions.md` |
| `--all` | All of the above | — | — |

**How it works:**
1. Symlinks `skills/` → `{tool_dir}/skills/` (so all tools share the same skills)
2. Copies `AGENTS.md` → tool-specific filename (CLAUDE.md, GEMINI.md, etc.)
3. For monorepos, copies ALL AGENTS.md files (root + component-level)

### Step 4.2: Create skill-sync

Generate `skills/skill-sync/SKILL.md` and `skills/skill-sync/assets/sync.sh`.

The sync script:
1. Reads all `skills/*/SKILL.md` files
2. Extracts `metadata.scope` and `metadata.auto_invoke` from YAML frontmatter
3. Generates the `### Auto-invoke Skills` table
4. Updates each AGENTS.md file's Auto-invoke section

**Idempotency requirement**: Running sync.sh multiple times produces identical output.
The script REPLACES the Auto-invoke section between markers, not appends.

**Scope mapping**: The sync script must map scope values to AGENTS.md paths.
For this project, detect the module structure and generate the scope mapping
dynamically. Use the template from `assets/sync.sh` and customize the
`get_agents_path()` function.

## Rules

1. **ANALYZE first, GENERATE second.** Never generate skills from assumptions.
2. **AGENTS.md MUST be under 150 lines.** If it's over, you're putting too much detail — move it to skills.
3. **Skills contain REAL content.** Extract actual file paths, code patterns, commands from the project.
4. **Don't duplicate content.** If it's in a module skill, don't repeat it in AGENTS.md.
5. **AGENTS.md is the MAP, skills are the TERRITORY.** The map should be glanceable.
6. **Idempotent re-runs.** Phase 0 detects existing artifacts via signatures. Generated files are safe to update; hand-written files are NEVER touched.
7. **setup.sh must be Bash 3 compatible.** macOS ships Bash 3 by default.
8. **sync.sh must be idempotent.** Running it twice produces identical output.
9. **Skills follow Agent Skills open standard.** Proper frontmatter, proper structure.
10. **This skill generates knowledge, SDD manages process.** Never create SDD artifacts.
11. **Each skill MUST have `scope` and `auto_invoke` in metadata.** Without these, sync.sh can't generate Auto-invoke tables.
12. **Code snippets in skills must be from the ACTUAL codebase.** Read the files and extract real patterns.
13. **If the project is too small (<5 source files), skip module skills.** Just generate overview + AGENTS.md.
